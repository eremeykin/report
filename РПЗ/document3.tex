\chapter{Техническое задание}
\section{Введение}
Система информационной поддержки выбора условий обработки тонкостенных деталей
разрабатывается для использования при назначении режимов резания податливых
деталей. Такая система может быть использована для оперативной проверки
допустимости режимов резания, рассчитанных предварительно.
\section{Основания для разработки}
Основанием для разработки указанной системы является задание
кафедры РК-9 МГТУ им. Н.Э. Баумана. 
\section{Назначение разработки}
Функциональным назначением разработки является моделирование процессов
деформирования нежестких деталей при их обработке резанием. При этом исходные
данные моделирования могут изменяться в зависимости от расчетного случая. 
Эксплуатационным назначением разработки является оптимизация режимов обработки.
\section{Требования к программному продукту}
Разрабатываемый программный продукт должен отвечать следующим требованиям:
\begin{itemize}
  \item Обеспечить моделирование процесса обработки для заданных условий.
  \item Предоставить удобный пользовательский интерфейс для выбора исходных
  данных.
  \item Реализовывать независимость средств управления моделированием от
  конкретных расчетных случаев.
  \item Гарантировать стабильную работу в условиях ошибочного ввода.
  \item Продукт должен иметь структуру, пригодную для дальнейшего расштрения.
\end{itemize}
\section{Стадии и этапы разработки}
Программный продукт разрабатывают в соответствии со следующими стадиями:
\begin{enumerate}
	\item Подготовка информации
	\item Разработка функциональной модели системы
	\item Разработка структуры системы
	\item Проектирование интерфейса
	\item Программирование продукта
	\item Тестирование, отладка
\end{enumerate}
\section{Порядок конторля и приемки}
Контроль продукта производится путем сравнительного анализа результатов
моделирования и введенных исходных данных. В качестве пробной модели используют
модель ANSYS, полученную на предыдущих этапах проработки темы. 

\chapter{Подготовка информации}
\section{Анализ проблемы}
В соответствии с поставленными задачами особенно следует отметить предполагаемую
независимость продукта от исходных данных. Эта особенность позволит в будущем
на основе разработанной системы создать экспертную систему, которая предоставит
средства для компьютеризированной генерации готовых решений. 
Именно с этой целью также планируется спроектировать удобную для
дальнейшего расширения программную структуру продукта. Такая структура
накладывает определенные ограничения на выбор программных средств. Следует
отказаться от известных решений возникающих проблем, если такие решения могут
негативно сказаться на гибкости продукта.

\section{Общие принципы разработки}
Современные принципы разработки сложных программных систем в большинстве случаев
подразумевают объектный подход к проблеме. Это связано прежде всего с тем, что
объектное представление позволяет разработчику оперировать понятиями из
реального мира. Еще одна выгодная сторона этого приёма - возможность
абстрагироваться от реализации модулей, которые в рамках выделенной подзадачи не
обладают первостепенной важностью. В совокупности разработчик программных систем
получает эффективное средство решения задачи. 

Большинство современных языков программирования поддерживают проектирование
систем в терминах объектного подхода. Следовательно, для всестороннего
использования возможностей ООП следует начинать проектирование системы уже с
учетом её объектной структуры. Для этого наилучшим образом подходит UML
моделирование. Стандарты UML хорошо соответствуют конструкциям объектных языков
программирования. Это позволит после окончательной спецификации системы
средствами UML реализовать структуру в программном коде с высокой степенью
соответствия задуманным представлениям.
\section{Выбор программных средств}
\subsection{Выбор системы моделирования}
Предыдущие этапы проработки темы показали, что для моделирования предметной
области наилучшим образом подходит система конечно-элементного расчета ANSYS. Её
применение в данном проекте обусловлено доступностью использования всех
функциональных возможностей системы без необходимости огранизации сложных
межпрограммных связей. Программа ANSYS позволяет составлять APDL скипты. Такие
скрипты как правило сохраняются в текстовых .mac файлах. После подготовки
скриптов возможно их исполнение в пакетном режиме. Это реализуется путем вызова
исполняемого файла ANSYS с параметром командной строки, соответствующим пути
скриптового файла. Также есть возможность анализировать результаты работы
ANSYS. Для этого в параметрах вызова надо указать выходной файл. Если
потребуется некоторое графическое отображение хода моделирования, то следует
в APDL скрипте запрограммировать вывод в один из графических форматов для
дальнейшего отображения.

Исходные данные и модель, подготовленные на этапе
исследования технологического деформирования в системе
ANSYS, позволят без дополнительных затрат оценить и отладить возможности
проектного решения.Эти данные можно использовать для первичного наполнения системы.

К отрицательным сторонам применения ANSYS в качестве расчетного средства
следует отнести б\textit{о}льшую сложность составления модели. Грамотное
составление модели в системе ANSYS требует навыков написания ADPL скриптов и
глубоких фундаментальных знаний предметной области. Тем не менее в соответствии
с техническим заданием предполгается распредлить функционал системы на
функционал пользователя и функционал разработчика. Такое решение позволит
полностью отстранить пользователя от работы со внутренними механизмами моделей.
Это означает что необходимо лишь разовая разработка универсальной
параметризованной модели для соответствующего расчётного случая. Такие модели
будут многократно вызываться с различными значениями параметров, следовательно
трудоемкость работы с системой в целом останется невысокой.

\subsection{Выбор языка программирования}
На сегодняшний день создано большое количество различных языков
программирования. Для того чтобы создать высококачественный программный продукт
наиболее целесообразно использовать один из высокоуровневых языков. Такие языки
позволяют быстро разрабатывать сложные системы при минимальных затратах времени.
Наиболее подходящими языками для решения поставленных задач при заданных
ограничениях представляются языки Java и C\#. 

Окончательно был выбран язык Java. Он полностью
отвечает требованиям разработки. Известно, что Java обеспечивает
кросплатформенность программ. Также к преимуществам разработки на Java можно
отнести автоматическое управление памятью, большой выбор готовых библиотек, как
для создания интерфейса, так и различных специализированных назначений. Он
широко распространен, поэтому поиск решения проблем, касающихся реализации, не
составит никакого труда.

\chapter{Разработка функциональной модели системы}
\section{Средства моделирования}
На начальном этапе проектирования системы следует
составить формализованное представление требований к системе. В соответствии с определенными общими принципами разработки для описания системы
применяют UML модели. Язык UML определяет для этого диаграммы
вариантов использования (Use Case Diagram). Построение диаграммы вариантов
использования проводят в следующем порядке: определяют границы системы,
указывают действующих лиц, обозначают варианты использования.
\section{Границы системы}
Система представляет собой программный комплекс автоматизированного
моделирования. Она включает в себя одну или несколько моделей, подсистему
ввода и вывода информации (графический интерфейс) а также подсистему разработки
моделей. Такого описания границ системы достаточно для построения диаграммы
вариантов использования, но оно несомненно потребует уточнения в дальнейшем.
Язык UML тербует указания названия системы. Для определенности и краткости
примем название "AnsExp".
\section{Действующие лица}
Действующие лица имеют сосредоточенные цели. Предварительный анализ технического
задания определил, что необходимо реализовать отдельное построение моделей и их
дальнейшее применение. По этому принципу в случае системы AnsExp можно выделить
всего два действующих лица: разработчик моделей и пользователь системы.
Соответственно разработчик моделей имеет цель пополнения системы своими знаниями
в то время как пользователь извлекает из системы данные, полученные на основе
составленных моделей.
\section{Варианты использования}
Определяют варианты использования для действующего лица "разработчик моделей".
К ним относят:
\begin{itemize}
	\item Создать модель
	\item Изменить модель
\end{itemize}
Тогда к вариантам использования, связанных с лицом "пользователь"\,относят:
\begin{itemize}
	\item Загрузить модель
	\item Ввести параметры модели
	\item Рассчитать параметры модели
	\item Рассчитать модель в ANSYS
	\item Посмотреть результаты расчета
\end{itemize}
\section{Понятие "модель"\, в терминах проектируемой системы}
По заготовленным данным для построения диаграммы вариантов использования видно,
что без дополнительной информации построение диаграммы не имеет смысла, т.к.
такая диаграмма не сможет в допустимой мере детализации описывать систему.
Это укрупненное представление дает мало информации о проектируемой системе,
поэтому следует на текущем этапе определить какой смысл следует вкладывать в
понятие "модель"\, в рамках поставленной задачи. 

Моделью прежде всего является сущность, которая может интерпретироваться
системой ANSYS. Т.е. это скриптовый .mac файл. С другой стороны пользователь
должен быть изолирован от системы расчета в ANSYS, следовательно, скриптовый
файл должен описывать только основные закономерности предметной области без
конкретизации параметров. 

Заполнение параметров может происходить по одному из
двух возможных сценариев: либо ввод строкового представления параметра, либо
выбор параметра из базы допустимых значений. Базу допустимых значений с
программной точки зрения удобно содержать в базе данных. Т.е. база данных -
второй необходимый элемент модели. 

Некоторые параметры в сложных моделях не
могут быть выбраны либо введены указанными способами, но они все равно должны
быть заполнены. Тогда необходимо реализовать промежуточные расчетные
механизмы. Такие параметры могут быть рассчиатны по этим механизмам. Расчетные
механизмы - третий компонент модели. В дальнейшем они будут называться
калькуляторами параметров.

В предметной области все параметры находятся в каких-либо отношениях между
собой. Система этих параметров и отношений между ними может быть представлена в
виде некоторой структуры. Структура параметров очень важна для сохранения
связи система - модель системы. Поэтому она должна сохраняться и в программном
виде. Следовательно, четвертый компонент - описание структуры параметров.

%  расширить вариант использования "Создать модель", включив в него
% более подробное описание. На диаграмме UML такие связи обозначаются "include". 
\section{Детализация диаграмы вариантов использования}
Итак, вариант использования "Создать модель"\, с учетом положений, описанных в
предыдущем разделе, можно детализировать разбиением на составляющие. Таких
составляющих всего четыре: 
\begin{itemize}
	\item Создать структуру модели
	\item Создать калькулятор параметров моделей
	\item Создать базу данных (БД) модели
	\item Создать расчетный шаблон модели
\end{itemize}

Нотация UML подразумевает использование связей с обозначением <<include>> для
указания включенных действий.

После определения понятия модель вариант использования "Ввести параметры
модели"\, можно уточнить указав два альтернативных включенных варианта:
\begin{itemize}
	\item Выбрать параметр из базы данных
	\item Ввести строковое предстваление параметра
\end{itemize}
Итак, теперь диаграмма вариантов использования обладает достаточной полнотой.

\chapter{Разработка структуры системы}
\section{Основные положения}
Диаграмма вариантов использования, разработанная на предыдущем этапе, позволила
определить и формализовать основные требования к функционалу системы. На
следующем этапе необходимо разработать структуру системы, ориентированную на
выполнение указанных требований. Придерживаясь выбранной нотации UML для
проектирования структуры используют диаграммы классов (Class Diagram). Диаграммы
классов на следующих этапах могут быть переведены в программный код Java без
существенных отклонений от спроектированной логики.

Отдельной проработки заслуживает структура модели. Она в целом уже была
описана на этапе функционального проектирования. Для визуализации описания
строят диаграмму компонентов. Диаграмма компонентов модели во многом определяет
структкру самой системы.
\section{Диаграмма компонентов модели}
Модель представляет собой независимую сущность, которая несет в себе полный
набор информации, касающейся одного расчетного случая. Как было сказано модель
включает в себя четыре функциональных компонента.
